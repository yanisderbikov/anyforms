# anyforms
backend for anyforms.ru

## Salesbot drip-кампания (триггер SalesBot по расписанию)

Сервис по расписанию запускает SalesBot'ы в amoCRM для сделок (лидов) в заданном
статусе воронки. На каждом слоте расписания лиду уезжает следующий по порядку бот.
Пока лид в целевом статусе — цепочка идёт; ответил клиент → amoCRM переносит сделку в
другой статус → лид выпадает из выборки и боты больше не уходят.

### Схема БД

Таблицы создаёт **Flyway** (`src/main/resources/db/migration/`):
- `V1__init_salesbot_schema.sql` — исходная схема (4 таблицы);
- `V2__salesbot_refinements.sql` — `schedule.time_utc` → `timestamptz` (под `Instant`) + сид расписания.

| Таблица | Назначение | Кто заполняет |
|---|---|---|
| `order_type_funnel` | тип заказа → воронка/статус amoCRM | **вручную** |
| `bot_sequence` | порядок ботов для каждого типа | **вручную** |
| `schedule` | слоты каденции (когда запускать), общие для всех типов | сид + **правки вручную** |
| `bot_execution_log` | журнал отправок (источник истины о прогрессе) | **приложение само** |

> Все времена в БД — в **UTC**. МСК = UTC+3 (без перехода на лето).
> `type` хранится строкой (enum `OrderType`): `RETAIL`, `RETAIL_REPEAT`, `CUSTOM`, `CUSTOM_REPEAT`.

---

### 1. `order_type_funnel` — где искать лидов

Одна строка на каждый тип, который надо гонять ботами. Отсюда шедулер берёт
`pipeline_id` / `status_id` для выборки лидов (запрос №1).

| Колонка | Что заполнять | Откуда брать |
|---|---|---|
| `type` | один из `RETAIL` / `RETAIL_REPEAT` / `CUSTOM` / `CUSTOM_REPEAT` (регистр важен) | enum `OrderType` |
| `pipeline_id` | ID воронки amoCRM | настройки amo / `GET /api/v4/leads/pipelines` / URL воронки |
| `status_id` | ID целевого статуса в этой воронке | оттуда же |
| `id` | не трогать | автоинкремент |

- `UNIQUE(type)` — на тип ровно одна пара воронка/статус.
- Нет строки для типа → тип не обрабатывается.

```sql
INSERT INTO order_type_funnel (type, pipeline_id, status_id) VALUES
  ('RETAIL',        9846162, 142),
  ('RETAIL_REPEAT', 9846162, 143),
  ('CUSTOM',        9846162, 144),
  ('CUSTOM_REPEAT', 9846162, 145);
```

---

### 2. `bot_sequence` — цепочка ботов

Одна строка = один бот на одной позиции для одного типа. Это и есть дрип-цепочка.

| Колонка | Что заполнять | Примечание |
|---|---|---|
| `type` | тот же enum `OrderType` | для какого типа цепочка |
| `position` | `1, 2, 3, …` | с 1, по возрастанию = порядок отправки |
| `bot_id` | ID SalesBot в amoCRM | из редактора SalesBot |
| `id` | не трогать | автоинкремент |

- `UNIQUE(type, position)` — позиции в типе не повторяются.
- На каждом слоте лиду уезжает **первая позиция без записи `SUCCESS`** в логе.
- Добавил новую старшую позицию (была 9 — стала 10) — она автоматически станет
  «следующей» для всех, у кого 1–9 уже `SUCCESS`.

```sql
INSERT INTO bot_sequence (type, position, bot_id) VALUES
  ('RETAIL', 1, 1001),
  ('RETAIL', 2, 1002),
  ('RETAIL', 3, 1003),
  ('CUSTOM', 1, 2001),
  ('CUSTOM', 2, 2002);
```

---

### 3. `schedule` — когда запускать (слоты, общие для всех типов)

Засеяна миграцией `V2`. Правится **без передеплоя**. Каждый слот = один шаг каденции
для всех лидов во всех целевых статусах.

| Колонка | Что заполнять | Примечание |
|---|---|---|
| `weekday` | день недели ISO: `1`=Пн … `7`=Вс | |
| `time_utc` | базовое время суток в **UTC** как `timestamptz` | значимо только **время суток**; дату ставь плейсхолдером `1970-01-01` |
| `enabled` | `TRUE` / `FALSE` | выключить слот, не удаляя |
| `id` | не трогать | автоинкремент |

Важное:
- **Время в UTC.** Из МСК вычитай 3 часа (Пн 10:38 МСК = `07:38` UTC).
- На рантайме добавляется **джиттер ±`salesbot.jitter.max-minutes` минут** (сейчас 30),
  детерминированный на сутки — фактический запуск чуть «плавает». Держи базовые времена «некруглыми».
- Тикер каждые `salesbot.scheduler.tick-ms` (сейчас 30000 мс) срабатывает на первом тике
  **в момент или после** (время слота + джиттер) — catch-up: пропущенная из-за рестарта/
  редкого тика минута не теряет слот. За сутки слот срабатывает один раз.

```sql
-- значимо только время суток в UTC; дата всегда 1970-01-01
INSERT INTO schedule (weekday, time_utc, enabled) VALUES
  (1, TIMESTAMPTZ '1970-01-01 07:38:00+00', TRUE),  -- Пн ~10:38 МСК
  (3, TIMESTAMPTZ '1970-01-01 14:06:00+00', TRUE);  -- Ср ~17:06 МСК

-- выключить слот:
UPDATE schedule SET enabled = FALSE WHERE id = 4;
```

Текущий сид (UTC ← МСК−3): Пн `07:38`, Ср `14:06`, Чт `12:23`, Пт `16:47`, Вс `09:14`.

### 4. `bot_execution_log` — заполняется автоматически, руками не трогать

Ведёт приложение. Источник истины о прогрессе цепочки — по нему вычисляется
«следующий бот». Только для чтения/диагностики.

| Колонка | Смысл |
|---|---|
| `lead_id` | сделка amoCRM |
| `bot_id` / `position` / `type` | какой бот / позиция / тип отработали |
| `status` | `SUCCESS` (запрос на запуск ушёл) или `FAILED` (лид вышел из статуса / ошибка запроса) |
| `date_executed` | момент попытки (`timestamptz`, UTC) |

- `UNIQUE(lead_id, bot_id)` — жёсткий бэкстоп от двойной отправки; запись через upsert.
- Прогресс = **первая позиция без `SUCCESS`**. `FAILED` прогресс не двигает (может позже
  «повыситься» до `SUCCESS`). Вся цепочка в `SUCCESS` → лиду больше ничего не шлётся.

---

### Минимальный чек-лист запуска

1. Заполнить `order_type_funnel` (типы → воронка/статус).
2. Заполнить `bot_sequence` (типы → боты по порядку).
3. Проверить/поправить `schedule` (времена в UTC).
4. `bot_execution_log` оставить пустым — заполнится сам.

### Конфигурация (`application.properties`)

| Свойство | Назначение |
|---|---|
| `salesbot.scheduler.enabled` | вкл/выкл тикер |
| `salesbot.scheduler.tick-ms` | период тика, мс |
| `salesbot.jitter.max-minutes` | максимальный джиттер времени слота (±N минут) |
| `salesbot.lock.key` | ключ Postgres advisory-lock (single-flight: один прогон одновременно во всём кластере) |

> Сборка — только под **JDK 21** (дефолтный `mvn` на JDK 25 ломает Lombok).
